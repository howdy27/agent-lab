## 트러블 슈팅 - 키파일 경로 설정 문제

### 문제 상황
앱 실행 시 2단계에서 FAIL 발생

### 가설

1. t_secret.key 파일 내용이 문제
2. 키파일 경로 설정(= 환경변수 설정) 문제

### 확인

1. t_secret.key 파일 내용 확인
```bash
cat $AGENT_HOME/api_keys/t_secret.key 
agent_api_key_test
```
정상적으로 입력되어 있음.


2. 키파일 경로 설정 문제
```bash
echo $AGENT_KEY_PATH
/home/agent-admin/agent-app/api_keys/t_secret.key
```
앱의 2단계 프로세스 에러 메세지를 살펴보면
Expected: /home/agent-admin/agent-app/api_keys
이지만, AGENT_KEY_PATH 설정은 t_secret key 까지 되어있음.

### 해결

bashrc 파일을 수정해주고

```bash 
cat > ~/.bashrc << 'EOF'
export AGENT_HOME=/home/agent-admin/agent-app
export AGENT_PORT=15034
export AGENT_UPLOAD_DIR=$AGENT_HOME/upload_files
export AGENT_KEY_PATH=/home/agent-admin/agent-app/api_keys
export AGENT_LOG_DIR=/var/log/agent-app
EOF
```

source 명령어를 통해 SSH 재접속 없이 변경사항 즉시 적용
```bash
 source ~/.bashrc
 echo $AGENT_KEY_PATH
/home/agent-admin/agent-app/api_keys
```

## 학습 목표 정리

### 1. SSH 포트 변경과 Root 원격 접속 차단이 왜 기본 보안에 해당하는지

자동화 봇은 기본 포트인 22번만 스캔하는 경우가 많다. 포트를 20022로 변경하면
봇의 스캔 대상에서 제외되어 자동화 공격 시도 자체가 줄어든다.

Root 계정은 서버의 모든 권한을 가지고 있어, 공격자가 Root로 직접 로그인에
성공하면 서버 전체가 장악된다. Root 없이도 sudo를 통해 충분히 개발과 유지보수가
가능하므로, 최소 권한 원칙을 실현하기 위해 차단한다.

### 2. UFW로 필요 포트만 허용하는 방화벽 정책 구성과 검증

UFW는 Ubuntu에 기본으로 포함된 방화벽 관리 도구다.
기본적으로 들어오는 트래픽은 전부 차단하고, 필요한 포트만 열어주는 방식이다.
나가는 트래픽은 서버가 패키지 업데이트, 외부 API 호출 등 정상적인 동작을
해야 하므로 전부 허용한다.


### 3. 역할 기반 계정/그룹과 ACL로 공유/보안 디렉토리를 분리하는 이유

최소 권한 원칙을 실현하기 위함이다. 불필요한 정보에 접근하는 것을 막고,
역할에 따라 접근할 수 있는 공간을 분리한다.

- upload_files → agent-common (admin, dev, test 모두 접근 가능)
- api_keys, /var/log/agent-app → agent-core (admin, dev만 접근 가능)

Linux 기본 권한은 소유그룹이 1개뿐이라 세밀한 제어가 어렵다.
ACL을 쓰면 여러 그룹에 각각 다른 권한을 줄 수 있어 협업과 보안을 동시에 실현할 수 있다.


### 4. 환경 변수로 실행 환경을 고정하는 이유와 검증 방법

절대경로를 하드코딩하면 경로가 바뀔 때마다 코드를 수정해야 한다.
환경 변수를 쓰면 변수만 바꾸면 되므로 유지보수가 쉽고 가독성도 높아진다.
또한 개발/테스트/운영 환경마다 환경 변수만 다르게 설정하면
코드 수정 없이 동작하므로 재사용성이 높아진다.



### 5. 쉘 스크립트로 프로세스/포트/리소스 상태를 수집하고 로그로 남기는 흐름

monitor.sh가 실행되면 아래 순서로 동작한다:

1. Health Check → 앱 프로세스와 포트 15034 확인. 비정상이면 exit 1로 종료
2. 방화벽 상태 점검 → UFW 비활성이면 WARNING 출력 후 계속 진행
3. 자원 수집 → top, free, df 명령어로 CPU/메모리/디스크 사용률 수집
4. 임계값 경고 → CPU>20%, MEM>10%, DISK>80% 초과 시
5. 로그 기록 → 타임스탬프와 함께 monitor.log에 이어쓰기
6. 로그 용량 관리 → 10MB 초과 시 최대 10개 파일 유지

### 6. crontab으로 모니터링을 주기 실행시키고 로그 보존 정책이 왜 필요한지

실제 서비스는 24시간 운영되므로 사람이 상주하면서 매분 monitor.sh를
수동으로 실행할 수 없다. cron으로 자동화하면 사람 없이도 주기적으로 실행된다.

cron을 통해 로그를 매 분 남기면 디스크가 꽉 차서 서버 전체가 멈출 수도 있다.
오래된 로그는 자동으로 삭제하여 디스크 용량을 관리해야 한다.
